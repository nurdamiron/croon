# ТЗ: интеграция склада Alash → Kaspi (долгосрочная архитектура)

> Документ для нового чата в репозитории `croon.kz/frontend`.
> Цель: один остаток на товар во всех каналах, синхронизация Alash → Kaspi через динамический XML price-feed.
> Этот файл — полное ТЗ. Все упомянутые артефакты лежат в репо `~/SheetsAlashBa3arKaspi/`.

---

## 0. Контекст

Три канала продаж:
- **croon.kz** — мастер каталога. Next.js 14 + Prisma + Postgres (Neon `eu-north-1`). 1895 SKU.
- **kaspi.kz** — тестовый аккаунт `30233309` (логин `beks.aiten@mail.ru`), 14 активных офферов, заказов нет. Один товар может иметь N карточек на Kaspi (видим: SKU 826 имеет два Kaspi-оффера `121012404_271787064` и `155617325_500472011`).
- **ba3ar.kz** — отдельный Next-репо (создаётся с нуля), общая БД с Alash. Контент свой, SKU совпадает.

Проблема сейчас: остатки никак не синхронизированы. Можно продать один товар дважды.

Полная архитектура долгосрочного решения описана в `~/SheetsAlashBa3arKaspi/docs/inventory-architecture.md` — этот документ это её **первая фаза** (Kaspi feed на динамике), плюс конкретный план реализации.

## 1. Цель этого этапа

Поднять **динамический endpoint** `https://croon.kz/api/kaspi/feed.xml`, который при каждом запросе Kaspi:

1. Читает из БД активные Kaspi-офферы и их связь с нашими `Product`.
2. Считает доступный остаток (`qtyOnHand - qtyReserved`).
3. Отдаёт XML формата `kaspi_catalog`.

В Kaspi Merchant Cabinet URL фида указывается в разделе **Магазин → Загрузка прайс-листа → Автоматическая загрузка** (Kaspi сам забирает раз в 60 минут).

После этого этапа:
- ✅ Остатки на Kaspi обновляются автоматически (≤1 час лаг).
- ✅ Появление новой Kaspi-карточки одного и того же товара = одна запись в админке, не правка XML.
- ✅ Цены меняются из админки.
- ❌ Заказы Kaspi всё ещё подтверждаются вручную в кабинете (это следующая фаза).
- ❌ Резервирование при заказе на сайте ещё не реализовано (тоже следующая фаза).

## 2. Изменения в Prisma

Минимальные. Полная схема в `~/SheetsAlashBa3arKaspi/docs/inventory-architecture.md`, но **на этой фазе делаем только**:

### 2.1. Расширение `Product`

```prisma
model Product {
  // ... всё что уже есть ...

  qtyOnHand    Int  @default(0)   // мигрируется из totalStock
  qtyReserved  Int  @default(0)   // 0 пока резервирование не подключено

  kaspiOffers  KaspiOffer[]
}
```

Миграция данных (внутри той же миграции, SQL после AutoGenerated):
```sql
UPDATE "Product" SET "qtyOnHand" = "totalStock";
```

`totalStock` пока **не удаляем** — оставляем как deprecated на время переходного периода. Удалим после Фазы 1 склада.

### 2.2. Новая модель `KaspiOffer`

```prisma
model KaspiOffer {
  id           String   @id @default(cuid())
  kaspiSku     String   @unique             // "557500965" или "121012404_271787064"
  productId    String                       // FK Product.id (наш мастер-SKU)
  kaspiStoreId String                       // "30233309_PP1"
  priceTenge   Int                          // цена в тенге, целое (Kaspi не принимает дробные)
  cityId       String   @default("750000000") // Костанай; список см. exports/kaspi-cities.json
  active       Boolean  @default(true)
  kaspiName    String?                      // как Kaspi назвал — для отладки
  kaspiBrand   String?
  lastSyncedAt DateTime?
  createdAt    DateTime @default(now())
  updatedAt    DateTime @updatedAt

  product      Product  @relation(fields: [productId], references: [id], onDelete: Cascade)

  @@index([productId])
  @@index([active])
}
```

**Почему так:**
- `kaspiSku` — уникальный, по нему Kaspi матчит наши офферы со своими карточками.
- `productId` без unique — один Product может иметь несколько `KaspiOffer` (3-4 карточки на товар).
- `priceTenge` хранится **на Kaspi-оффере**, а не на `Product`, потому что на Kaspi цены отличаются от сайта (комиссия маркетплейса, акции). Цена на сайте — это `Product.price` как сейчас.
- `cityId` пока всегда `750000000` (Костанай). В будущем — расширение на города через отдельную модель `KaspiOfferCityPrice` если понадобится разная цена по городам.

### 2.3. Команды миграции

```bash

npx prisma migrate dev --name add_kaspi_offer_and_stock_fields
```

## 4. Endpoint `/api/kaspi/feed.xml`

Файл: `frontend/src/app/api/kaspi/feed.xml/route.ts`

### Контракт

- **Метод:** GET
- **Auth:** опционально HTTP Basic (см. секцию 5). Без auth — открытый endpoint (Kaspi так может).
- **Ответ:** `Content-Type: application/xml; charset=utf-8`, `Cache-Control: no-store`
- **Тело:** XML по схеме `kaspiShopping`.

### Алгоритм

1. `SELECT * FROM "KaspiOffer" WHERE active = true` + join `Product` (`name`, `qtyOnHand`, `qtyReserved`).
2. Для каждого оффера:
   - `stockCount = max(0, qtyOnHand - qtyReserved)`
   - `available = stockCount > 0 ? "yes" : "no"`
3. Сборка XML строкой (без библиотек — нужна минимальная зависимость).

### Эталон формата

Шаблон один в один как `~/SheetsAlashBa3arKaspi/exports/kaspi-active-original.xml`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<kaspi_catalog xmlns="kaspiShopping" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://kaspi.kz/kaspishopping.xsd" date="2026-05-18 17:30">
    <company>30233309</company>
    <merchantid>30233309</merchantid>
    <offers>
        <offer sku="557500965">
            <model>Модуль расширения Raspberry Pi Edge TPU Google Coral</model>
            <brand>raspberry-pi</brand>
            <availabilities>
                <availability available="yes" storeId="30233309_PP1" preOrder="0" stockCount="2"/>
            </availabilities>
            <cityprices>
                <cityprice cityId="750000000">150000</cityprice>
            </cityprices>
        </offer>
        <!-- ... остальные офферы ... -->
    </offers>
</kaspi_catalog>
```

### Готовый код (положить как есть в `route.ts`)

```ts
import { NextResponse } from 'next/server';
import { prisma } from '@/lib/prisma'; // или путь который у вас принят

export const dynamic = 'force-dynamic';
export const revalidate = 0;

const MERCHANT_ID = process.env.KASPI_MERCHANT_ID || '30233309';

function esc(s: string | null | undefined): string {
  return (s ?? '').replace(/&/g, '&amp;').replace(/</g, '&lt;')
                  .replace(/>/g, '&gt;').replace(/"/g, '&quot;');
}

function pad(n: number): string { return String(n).padStart(2, '0'); }

export async function GET() {
  const offers = await prisma.kaspiOffer.findMany({
    where: { active: true },
    include: {
      product: { select: { name: true, qtyOnHand: true, qtyReserved: true } },
    },
    orderBy: { createdAt: 'asc' },
  });

  const now = new Date();
  const date = `${now.getFullYear()}-${pad(now.getMonth()+1)}-${pad(now.getDate())} ${pad(now.getHours())}:${pad(now.getMinutes())}`;

  const offerXml = offers.map(o => {
    const stock = Math.max(0, o.product.qtyOnHand - o.product.qtyReserved);
    const yn = stock > 0 ? 'yes' : 'no';
    const name = esc(o.kaspiName || o.product.name).slice(0, 250);
    return `        <offer sku="${esc(o.kaspiSku)}">
            <model>${name}</model>
            <brand>${esc(o.kaspiBrand || 'Alash')}</brand>
            <availabilities>
                <availability available="${yn}" storeId="${esc(o.kaspiStoreId)}" preOrder="0" stockCount="${stock}"/>
            </availabilities>
            <cityprices>
                <cityprice cityId="${esc(o.cityId)}">${o.priceTenge}</cityprice>
            </cityprices>
        </offer>`;
  }).join('\n');

  const xml = `<?xml version="1.0" encoding="UTF-8"?>
<kaspi_catalog xmlns="kaspiShopping" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation="http://kaspi.kz/kaspishopping.xsd" date="${date}">
    <company>${MERCHANT_ID}</company>
    <merchantid>${MERCHANT_ID}</merchantid>
    <offers>
${offerXml}
    </offers>
</kaspi_catalog>`;

  return new NextResponse(xml, {
    status: 200,
    headers: {
      'Content-Type': 'application/xml; charset=utf-8',
      'Cache-Control': 'no-store',
    },
  });
}
```

## 5. Опционально: защита Basic Auth

Kaspi поддерживает Basic Auth для прайса (в форме "Авторизация (опционально) — Логин/Пароль").

Если хотим закрыть feed:

```ts
// в начале route.ts
const expectedUser = process.env.KASPI_FEED_USER;
const expectedPass = process.env.KASPI_FEED_PASS;

if (expectedUser && expectedPass) {
  const auth = request.headers.get('authorization');
  const expected = 'Basic ' + Buffer.from(`${expectedUser}:${expectedPass}`).toString('base64');
  if (auth !== expected) {
    return new NextResponse('Unauthorized', { status: 401, headers: { 'WWW-Authenticate': 'Basic' } });
  }
}
```

Тогда в форме Kaspi нужно прописать ту же пару `KASPI_FEED_USER:KASPI_FEED_PASS`.

**Решение по этой фазе:** **не закрываем**. URL непредсказуемый, content публично-безобидный (наши цены и так видны на Kaspi). Закрытие добавляет точку отказа — если перепутаем пароль, фид перестанет обновляться, никакого алерта от Kaspi.

## 6. Админка `/admin/kaspi-offers`

Минимум на этой фазе. Файл: `frontend/src/app/admin/kaspi-offers/page.tsx`.

### Что должна уметь

1. **Таблица** всех `KaspiOffer` с полями:
   - `kaspiSku` (read-only после создания)
   - `kaspiName` (read-only, для справки)
   - `Product.id` + `Product.name` (через select/search)
   - `priceTenge`
   - `kaspiStoreId`
   - `active` (toggle)
   - текущий `qtyAvailable = Product.qtyOnHand - Product.qtyReserved` (read-only из join)

2. **Добавить новый оффер** — форма:
   - kaspiSku (вводится вручную, из кабинета Kaspi)
   - выбор Product через поиск по `pg_trgm` (у вас уже есть)
   - priceTenge
   - kaspiStoreId
   - сохранить → `prisma.kaspiOffer.create`

3. **Удалить / деактивировать** — toggle `active`.

4. **Импорт из CSV** (для bulk-операций) — кнопка загрузки `kaspi-mapping.csv` + `ACTIVE.xml`. Делает то же что seed-скрипт.

Доступ — только `role === 'ADMIN'` (у вас уже есть в `User.role`).

## 7. Скрипт массовой синхронизации цен с Kaspi (опц.)

Когда Kaspi-кабинет — основной источник цен (например, после акций менеджер меняет цены на Kaspi), нужно тянуть `priceTenge` обратно из их выгрузки.

Скрипт `scripts/sync-prices-from-kaspi-export.js`:
- На вход — путь к скачанному из MC `Excel`/`XML`.
- Для каждого `kaspiSku` обновляет `KaspiOffer.priceTenge` в БД.

Запускается вручную после каждого изменения цен в MC. Можно оформить как `npm run sync:kaspi-prices`.

На этой фазе **не обязательно** — оставим на потом.

## 8. Проверка после деплоя

### Локально перед деплоем

```bash
cd frontend
npm run dev
# в другом терминале:
curl http://localhost:3000/api/kaspi/feed.xml | xmllint --format -
```

Ожидаем валидный XML с 12 офферами (или 14 если оставили чужие), правильными остатками.

### После деплоя

1. `curl https://croon.kz/api/kaspi/feed.xml` — должен отдать тот же XML.
2. В Kaspi MC → Магазин → **Загрузка прайс-листа → Автоматическая загрузка**:
   - URL: `https://croon.kz/api/kaspi/feed.xml`
   - Логин/пароль: пусто (или те что в env если закрыли auth).
   - **Проверить** → должно сказать "Найдено N офферов, ошибок 0".
   - Включить тумблер, **Сохранить**.
3. Через ~60 минут — в "Управление товарами" остатки 14 карточек должны совпасть с `Product.qtyOnHand` в БД.

### Алерты

В Telegram-канал (у вас уже есть `lib/telegram.ts`) слать когда:
- Endpoint падает (5xx за последние 5 минут).
- Kaspi-карточка `active=true` без привязки к существующему `Product`.
- Прошло > 90 минут с момента `lastSyncedAt` хотя бы у одного оффера (нужно проставлять при заборе Kaspi, см. секцию 9).

`lastSyncedAt` обновлять не получится через сам endpoint (мы же не знаем когда Kaspi его реально съел). Поэтому второй вариант: писать в отдельную таблицу `KaspiFeedHit { hitAt, userAgent, ip }` каждый раз когда фид запрашивают. Алерт = "прошло >90 мин без забора". Это **дополнительная мелкая модель**, делать только если нужен мониторинг.

## 12. Risks / открытые вопросы

1. **Имена офферов на Kaspi.** Когда модерация Kaspi переименовывает карточку, наш `kaspiName` устаревает. Решение: хранить в `kaspiName` как fallback, в `<model>` отдавать `Product.name`. **Сейчас в коде route.ts стоит именно так**: `kaspiName || product.name`. Если Kaspi откажется матчить из-за расхождения — переключить на `Product.name`.
2. **Один storeId, один cityId.** Если будут добавлены пункты выдачи в других городах — расширить `KaspiOffer` либо добавить `KaspiOfferStock { kaspiOfferId, storeId, cityId, stock }`. Сейчас не нужно.
3. **2 оффера на 826.** Оба будут получать `stock = 318` независимо друг от друга. Если оба продадут одновременно одну единицу — пока что списания нет (Reservation не в этой фазе), и это нормально. После фазы Reservation такая ситуация будет правильно блокироваться через `SELECT FOR UPDATE` на `Product`.
4. **Что если Kaspi-карточка снята с продажи в MC.** Она пропадает из `ACTIVE.xml`. Наш `KaspiOffer.active=true` останется, и мы будем продолжать слать её остаток. Это не вредит (Kaspi проигнорирует неизвестный sku), но засоряет фид. Решение позже: cron раз в день берёт у Kaspi `ACTIVE.xml` (его можно скачать из кабинета как мы делали), сравнивает с нашими `KaspiOffer.active=true`, недостающие — `active=false`.
