## 🔴 #1 — Двойная продажа последней штуки (сайт + Kaspi)
**Файл:** `src/lib/kaspi-sync.ts` → `applyItemTransition` (списание totalStock).
**Кейс:** 1 шт на складе, выложен и на сайте, и на Kaspi. Клиент A покупает на
сайте (сток 1→0, атомарно), в ту же секунду клиент B получает товар на Kaspi
(COMPLETED). Синк списывает totalStock-=1 БЕЗ проверки доступности →
`GREATEST(0,0-1)=0`, списание «прошло», но продали двоим при 1 шт.
**Причина:** сайт проверяет наличие перед списанием, синк — нет.
**Фикс:** атомарное условное списание (WHERE totalStock>=qty) + флаг/лог, если
не хватило, чтобы заметить конфликт.

## 🔴 #2 — Задвоение брони при сбое посреди заказа
**Файл:** `src/lib/kaspi-sync.ts:274-286` — stockApplied ставится ПОСЛЕ цикла по
позициям, изменения остатков — отдельными запросами вне транзакции.
**Кейс:** заказ из 3 товаров, оплачен. Синк бронирует товар1, товар2, на товаре3
падает (таймаут БД). Метка stockApplied не записана. Следующий синк считает
бронь не применённой → бронирует все 3 заново. Товары 1,2 забронированы дважды.
**Результат:** reservedStock завышен, доступный остаток занижен.
**Фикс:** применение остатков по заказу + установка stockApplied в ОДНОЙ
транзакции (prisma.$transaction).

## 🔴 #3 — Вечная бронь у брошенного заказа
**Кейс:** клиент оплатил на Kaspi (бронь), не забрал. Kaspi отменил/заархивировал,
но мы не словили статус (или заказ вне окна синка — см. #4). Бронь висит вечно,
товар «заморожен».
**Фикс:** таймаут на reserved (напр. снимать бронь у reserved-заказов старше N
дней, если их статус не подтверждён), или см. #4.

## 🟡 #4 — Старые заказы выпадают из окна синка
**Файл:** cron `SYNC_DAYS=14` (на EC2) + `syncKaspiOrders(daysBack)`.
**Кейс:** заказ оплачен 20 дней назад (reserved), сегодня возвращён (RETURNED).
Синк смотрит 14 дней → не видит → бронь не снимается.
**Фикс:** увеличить окно ИЛИ всегда дотягивать незакрытые reserved-заказы из БД
по их kaspiOrderId независимо от даты (точечный GET по заказу).

## 🟡 #5 — Защита от дубля карточки только в коде, не в БД
**Файл:** проверка в `api/admin/kaspi-offers/route.ts` (POST) и
`api/admin/products/[id]/kaspi-offers/route.ts` (PUT).
**Кейс:** два одновременных запроса привязки одного SKU к разным товарам обходят
проверку (товара ещё нет в момент обоих чтений). Или импорт мимо API.
**Фикс:** на уровне БД нет естественного уникального ключа (один товар может
иметь много карточек). Вариант: индекс по нормализованному kaspi product-id +
проверка в транзакции, либо периодическая проверка-«детектор дублей».

## 🟡 #6 — Позиция навсегда «непривязана» из-за разовой ошибки API
**Файл:** `src/lib/kaspi-api.ts` `getEntryProduct` (null при любой ошибке) +
`kaspi-sync.ts:235` (позиции тянутся только если items.length===0).
**Кейс:** при импорте заказа Kaspi отдал 429/таймаут на запрос товара позиции →
позиция создаётся с пустым kaspiSku. При след. синке позиции «уже есть» →
не перезапрашиваются. Заказ навсегда «непривязан», бронь не работает.
**Фикс:** перезапрашивать позиции с пустым kaspiSku при каждом синке (а не только
когда их нет), и/или не создавать позицию, если getEntryProduct вернул null.
