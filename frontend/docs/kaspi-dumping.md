# ТЗ: Каспи Демпинг (автоматическое управление ценой)

> Статус: **черновик / проектирование**. Автор-сборка: 2026-05-30.
> Обновлено 2026-05-30: добавлены §1.6 (верифицированные факты Kaspi API — офиц.доки
> + live-проверка offer-view + адверсариальная верификация) и §8.1 (антидемпинг-барьеры).
> Цель: встроить в админку автоматический репрайсер цен Kaspi (демпинг-бот),
> аналог сервиса AlgaTop, но на нашей инфраструктуре и нашем фиде.

---

## 0. Краткая суть

Бот периодически смотрит цены конкурентов по каждому товару на Kaspi и
автоматически держит нашу цену в нужной позиции (первое место / равная цена /
второе место), не опускаясь ниже заданного **floor (мин. цены)** и не поднимаясь
выше **ceiling (макс. цены)**. Магазин на первом месте в карточке получает ~90%
заказов — в этом весь смысл.

Всё реализуемо **серверно** (без браузера/прокси): Kaspi отдаёт список продавцов
по публичному JSON-эндпоинту, а цену мы меняем через наш существующий XML-фид.

## 1. Проверенные факты (фундамент ТЗ)

### 1.1 Источник цен конкурентов — РАБОТАЕТ с сервера

```
POST https://kaspi.kz/yml/offer-view/offers/{PID}
Content-Type: application/json
Referer: https://kaspi.kz/shop/p/-{PID}/
User-Agent: <обычный Chrome UA>

body: {"cityId":"750000000","id":"{PID}","page":0,"limit":12,
       "sortOption":"PRICE","installationId":"-1"}
```

Ответ (проверено живьём 2026-05-30):

```json
{
  "offers": [
    {"price":897.0,"merchantId":"30383258","merchantName":"ИП КРУН.kz",
     "merchantRating":4.7,"merchantReviewsQuantity":85,"kaspiDelivery":true,
     "merchantSku":"...","masterSku":"139531143"},
    {"price":899.0,"merchantId":"30435308","merchantName":"RadioMart.kz", ...}
  ],
  "total": 2,
  "offersCount": 2,
  "hasBetterOffer": false,
  "excellentMerchantPresent": false,
  "highRatingPresent": false
}
```

- `offers[]` **отсортирован по цене** (sortOption=PRICE), первый = лидер выдачи.
- Поля на оффер: `price`, `merchantId`, `merchantName`, `merchantRating`,
  `merchantReviewsQuantity`, `kaspiDelivery`, `merchantSku`, `masterSku`.
- `total` — всего продавцов; пагинация через `page`. `limit` до ~12 хватает
  (нам нужны только верхние позиции).
- `total: 0` → нас (и никого) нет в выдаче по этому городу = товар снят/недоступен.
- **Эндпоинт не блокирует серверные запросы** (в отличие от HTML-страниц карточек
  каталога — см. заметку `kaspi-catalog-urls.md`). Проверено с локалки; на EC2
  ожидается так же, **обязательно перепроверить при внедрении**.

### 1.2 Наш идентификатор продавца

- **В offer-view наш `merchantId` = `30383258`** (имя «ИП КРУН.kz»).
  Проверено на двух товарах (CJMCU-811, LM35DZ).
- **В XML-фиде** (`feed.xml/route.ts`) используется `KASPI_MERCHANT_ID = 30233309`
  и storeId `30233309_PP1`. **Это другой ID** (вероятно UID магазина vs merchantId
  выдачи). ⚠️ При реализации зафиксировать `30383258` как «свой» для исключения
  себя из списка конкурентов. Вынести в env `KASPI_OFFERVIEW_MERCHANT_ID`.

### 1.3 PID (Kaspi product id) у нас уже выводится

- `lib/kaspi-url.ts → kaspiPidFromSku(sku)`: SKU вида `PID_xxxxx` → `PID`.
- Где SKU «голый» — PID лежит в `KaspiCatalogEntry.kaspiProductId`.
- Эти же связи уже используются в `feed.xml` (маппинг offer→merchantSku).

### 1.4 Как мы ставим цену — через существующий фид

- Цена Kaspi живёт в **`KaspiOffer.priceTenge`** (Int, тенге).
- `feed.xml/route.ts` отдаёт её в `<cityprice>` по всем городам.
- **Цена Kaspi ИЗОЛИРОВАНА от `ProductVariant.price`** → демпинг НЕ трогает цены
  сайта / Satu / Ba3ar. Это ключевое преимущество нашей архитектуры: репрайсер
  меняет только `KaspiOffer.priceTenge`, остальные каналы не задеты.
- Менять цену = записать новый `priceTenge` → Kaspi подтянет с фида (стандартная
  скорость обновления Kaspi-фида, минуты).

### 1.5 Реальное состояние наших данных (важно для floor)

Из выгрузки AlgaTop (575 товаров нашего магазина):
- **costPrice (закуп) не заполнен почти нигде** («Без закуп цены 188»). Значит
  авто-floor от costPrice сейчас не сработает — нужно сначала залить закуп, либо
  на старте floor задавать вручную/через Excel.
- 188 товаров «на продаже», 387 «сняты с продаж».
- Шаг по умолчанию в AlgaTop = **2 ₸**.

## 1.6 Верифицированные факты Kaspi (исследование 2026-05-30)

> Собрано многоагентным исследованием: офиц. документация guide.kaspi.kz +
> live-проверка offer-view (curl) + адверсариальная верификация каждого
> критичного утверждения. Вердикты: CONFIRMED / UNCERTAIN / REFUTED.

### 1.6.1 Способ смены цены — официально подтверждён

- **[CONFIRMED]** Официального **REST-эндпоинта для смены цены НЕ существует.**
  Merchant API (X-Auth-Token) в разделе «Товары» умеет ТОЛЬКО импорт новых карточек
  (`POST /shop/api/products/import`). Менять цену выставленного товара через API
  нельзя. Источник: guide.kaspi.kz/partner/ru/shop/api/goods.
- **[CONFIRMED]** Официальные способы смены цены: (1) вручную в кабинете/приложении
  Kaspi Pay, (2) Excel-прайс вручную, (3) **XML-прайс по URL (наш фид)**. Для
  автоматизации единственный машинный путь — прайс-лист. Наш `KaspiOffer.priceTenge`
  → `feed.xml` = единственный официальный канал авто-демпинга. **Дизайн ТЗ верен.**
- **[CONFIRMED]** Цена в XML: целые числа, **с НДС**, без пробелов/разделителей.
  По-городовые цены (`<cityprice cityId=...>`) разрешены только в XML и только если
  склады в разных городах; в одном городе — единая цена. Excel-путь нам неприменим.
- **[CONFIRMED]** Два независимых канала: X-Auth-Token (API, только чтение заказов +
  импорт) ≠ Basic Auth (XML-фид по URL). Смена цены живёт ТОЛЬКО в фид-канале.
  Подтверждает архитектуру §2.0/§9.8.

### 1.6.2 Скорость демпинга — ГЛАВНОЕ ОГРАНИЧЕНИЕ

- **[CONFIRMED]** Kaspi перечитывает XML-фид по URL **раз в 60 минут** (только если
  содержимое изменилось) + **~15 мин** на применение. Источник:
  guide.kaspi.kz/partner/ru/shop/goods/price_list/q3251, q2962.
- ⚠️ **Реальная реактивность нашего демпинга через фид = до ~60–75 минут**, НЕ минуты
  и не секунды. Это потолок XML-канала.
- **[UNCERTAIN, важная поправка]** 60 минут — это интервал именно XML-фида, НЕ общий
  потолок. Коммерческие боты (FixBot «30 сек», PriceFeed «2 мин», AlgaTop-турбо «20 сек»)
  меняют цену быстрее **через имитацию действий в кабинете**, а НЕ через фид. Точный
  rate-limit Kaspi на смену цены публично НЕ задокументирован.
- 🔑 **Вывод для ТЗ:** мы сознательно НЕ лезем в кабинет (§2.0), значит наш демпинг
  работает в такте **~час**. Это нормально для большинства товаров. «Турбо» в нашем
  смысле = чаще обновлять *наш фид-файл* (он перечитывается раз в час всё равно) —
  суб-минутный демпинг как у AlgaTop нам недоступен без кабинетного канала. **В UI и
  SLA честно писать: «цена применится в течение часа-полутора».** Не обещать секунды.

### 1.6.3 Ранжирование Kaspi ≠ только цена

- **[UNCERTAIN]** Приоритет позиции в карточке (офиц. q3190): **1) бонусы продавца →
  2) цена → 3) рейтинг → 4) отзывы.** Цитата: «продавец с ценой ниже будет первым,
  даже если у других выше рейтинг и больше бонусы».
- **Поправка верификации:** бонусы работают как **эффективная цена** (цена − бонус),
  а не жёсткий верхний ярус. Конкурент с бонусом встаёт выше, только если его
  предложение выгоднее по совокупности. Чистый ценовой демпинг гарантирует топ ТОЛЬКО
  среди продавцов без бонусов / при равном бонусе.
- **Доставка тоже влияет:** `kaspiDelivery`, `deliveryDuration`, локальность
  (`interCity=false`), срок `delivery`. При близкой цене более быстрая/локальная
  доставка может вывести вперёд (см. живой кейс 116186418 в Приложении A).
- 🔑 **Вывод:** бот должен (а) детектировать бонусы конкурентов (если видны), (б) при
  отставании по доставке — опускать цену чуть ниже, при преимуществе — можно держать
  вровень. MATCH_FIRST имеет смысл, когда у нас лучше рейтинг ИЛИ доставка.

### 1.6.4 offer-view — полная спецификация (live-проверено)

- **[CONFIRMED]** Только **POST** (GET → 405). **Referer обязателен**
  (`https://kaspi.kz/shop/p/-{PID}/`, иначе 403). `Content-Type: application/json`
  обязателен (иначе 415). Все 6 полей тела обязательны
  (`cityId, id, page, limit, sortOption, installationId`) — иначе 400.
- **[CONFIRMED]** `limit` максимум **64** (≥65 → HTTP 400). Использовать `limit=64`;
  при >64 продавцов листать `page=0,1,2…` до пустого `offers[]`.
- **[CONFIRMED]** Пагинация: offset = `page*limit`, страницы непрерывны, единая
  сортировка. `total`/`offersCount` = число офферов по текущему запросу (после
  фильтров города). `total=0` / пустой `offers[]` при HTTP 200 = **нет конкурентов /
  товара нет в этом городе** (НЕ ошибка — удобно для пакетной проверки).
- **[CONFIRMED]** `sortOption` принимает только `PRICE` (по возрастанию, строго) и
  `RATING`. Для демпинга всегда `PRICE` → `offers[0]` = самый дешёвый. Сортировать на
  клиенте не нужно.
- **[CONFIRMED]** Город реально влияет на СОСТАВ офферов (Шымкент дал доп. продавца
  для iPhone 13). Мониторить по тому же `cityId`, где реально конкурируем (Костанай).
- **[CONFIRMED]** Стабилен: 6 запросов подряд — 200, без 429/каптчи, латентность
  ~0.44–0.83с. Для массового мониторинга держать паузу ~0.3–1с между запросами,
  распределять во времени (не burst на сотни PID).
- **[CONFIRMED]** Поля оффера: `price, merchantId, merchantName, merchantSku,
  merchantRating, merchantReviewsQuantity, kaspiDelivery, deliveryDuration, delivery,
  deliveryType, deliveryOptions{deliveryCost,deliveryThreshold,interCity}, preorder,
  locatedInPoint`. ⚠️ `merchantId` НЕ всегда числовой — бывает строковый slug
  (напр. `"Berekekz"`). Наш id — числовой `30383258`.
- **[UNCERTAIN]** Топ-флаги `hasBetterOffer`/`excellentMerchantPresent`/
  `highRatingPresent` в анонимном режиме НЕнадёжны (вероятно завязаны на авторизованную
  сессию). **НЕ опираться на них** — позицию вычислять самим: сравнивать наш
  `price`/`merchantId` с `offers[0]`.

### 1.6.5 Идентификатор магазина — гипотеза по рассинхрону

- **[MEDIUM]** `data-merchant-code` в виджете кредита = «Идентификатор магазина из
  кабинета (Настройки → Идентификатор магазина)». Вероятно это наш `30233309`
  (фид/кабинет), а `30383258` — отдельный `merchantId` выдачи offer-view.
- 🔑 Для исключения себя из конкурентов брать **`30383258`** (offer-view), НЕ
  merchant-code. Гипотезу подтвердить, заглянув в Настройки→Идентификатор магазина.

## 2. Эталон: как это устроено у AlgaTop (конкурент)

Изучён их лендинг + реальный кабинет, подключённый к нашему магазину, + Excel-экспорт.

### 2.0 Как AlgaTop получает доступ к магазину (и почему НАМ это НЕ нужно)

AlgaTop — внешний сервис, поэтому он **логинится в личный кабинет продавца Kaspi
от имени клиента**. При «Добавить магазин» предлагают два способа:
- **Email + пароль** от кабинета Kaspi (они называют «надёжнее»);
- **Телефон + SMS-код** (плашка-предупреждение: *«не обеспечивает 100% надёжность
  работы бота»* — т.е. даёт урезанные права/сессию).

Прямая цитата из их формы: *«Данные кабинета нужны для корректной работы сервиса —
**изменения цен и получения списка ваших товаров на Kaspi**»*. После входа —
*«идёт синхронизация товаров, 30с–3мин»*, появляется раздел **«Накладные»**.

Вывод: они работают через **внутренний кабинетный API Kaspi** (mc.shop.kaspi.kz /
кабинет продавца) под авторизованной сессией клиента — тянут весь каталог
(включая «снятые с продаж» и накладные → закуп), и, вероятно, меняют цены тоже
через кабинет. Отсюда их боль: хранение чужого пароля, поломки при смене
пароля/2FA, «ненадёжный» телефонный вход.

> ⛔ **НАМ ЭТОТ МЕХАНИЗМ НЕ НУЖЕН.** Мы — сам магазин, у нас уже есть:
> - официальный **`X-Auth-Token` Merchant API** (`lib/kaspi-api.ts`) — заказы;
> - **собственный XML-фид** — легальный канал смены цены (поменять цену = записать
>   `KaspiOffer.priceTenge`, БЕЗ логина в кабинет);
> - **собственный `costPrice`** в нашей БД — не нужно парсить накладные;
> - **собственный список товаров/офферов** (`KaspiOffer`/`KaspiCatalogEntry`).
>
> Единственное, что берём у Kaspi «снаружи» — **публичный `offer-view`** (цены
> конкурентов), который **не требует авторизации**. Поэтому в нашей реализации
> НЕ делаем «вход в кабинет по email/SMS», не храним пароль продавца и не
> копируем схему AlgaTop с авторизованной сессией. Это и проще, и безопаснее, и
> надёжнее их подхода.

### 2.1 Стратегии управления ценой (6 у них)

1. **Стать первым (демпинг)** — цена = цена 1 места − шаг. ← реализуем
2. **Цена первого места** — равная цене лидера (ставка на наш рейтинг: при равной
   цене Kaspi ставит выше того, у кого рейтинг лучше). ← реализуем
3. **Прижиматься к первому (2 место)** — сознательно держать 2-е место.
4. **Конкуренция внутри города** — выбор городов (`cityId`).
5. **Неконкуренция с магазинами** — список merchantId, кого не обгонять. ← реализуем
6. **Борьба за 2-20 место** — дойдя до floor, бороться за лучшее из оставшихся мест.

### 2.2 Скорость

- Стандарт: обновление цен < 3 мин (80% ассортимента).
- «Турбо»: до 20 сек, до 3 раз/мин — только для ключевых товаров (платно у них).
- Прямо предупреждают: **Kaspi ограничивает частоту обновления цен** — турбо
  только на важные позиции, иначе лимит.

### 2.3 Поля на товар (из их Excel-экспорта — 17 колонок)

```
Магазин | Артикул | Название | Статус | Город | Цена | Цена 1 места |
Цена закупа | Автоснижение | Мин.цена | Автоповышение | Макс.цена |
Шаг демпинга | Позиция | Остатки | Предзаказ | Артикул Каспи
```

Ключевое: **два независимых тумблера** — «Автоснижение» и «Автоповышение»
(не один enabled). Можно только ронять (держать топ), только поднимать
(когда конкурент ушёл — вернуть маржу), или оба.

### 2.4 Фильтры в их кабинете (карта состояний)

- **Статус**: На продаже / Сняты с продаж
- **Демпинг**: Автоснижение вкл/выкл, Автоповышение вкл/выкл
- **Мин/макс цены**: «Не хватает мин цены» (демпинг вкл, floor не задан — ОПАСНО),
  С/Без мин цены, С/Без макс цены
- **Место**: На первом месте / Ниже 1 места
- **Конкуренция**: Без конкурентов / С конкурентами
- **Закуп. цена**: С/Без закуп. цены
- **Предзаказ**: С/Без предзаказа

### 2.5 Расписание бота (advanced)

Зелёный баннер: «расписание работы бота» — ночью выключать, утром включать с
макс. ценой (одна настройка = выше маржа каждый день). В MVP не делаем, но поле
под расписание заложить.

## 4. Модель данных

Расширяем существующую модель **`KaspiOffer`** (там уже живёт `priceTenge`).
Применять через `ALTER TABLE ADD COLUMN IF NOT EXISTS` по SSM (см. примечание про
`prisma db push --accept-data-loss` в корневом CLAUDE.md).

```prisma
model KaspiOffer {
  // ...existing fields...

  // --- Демпинг: настройки ---
  autoDownscale   Boolean  @default(false)  // тумблер «Автоснижение»
  autoUpscale     Boolean  @default(false)  // тумблер «Автоповышение»
  minPriceTenge   Int?                       // floor — ниже не падать
  maxPriceTenge   Int?                       // ceiling — выше не подниматься
  dumpingStep     Int      @default(2)       // шаг (тенге)
  strategy        String   @default("BECOME_FIRST") // BECOME_FIRST|MATCH_FIRST|HOLD_SECOND
  ignoreMerchants String[] @default([])      // merchantId, кого не обгонять
  turbo           Boolean  @default(false)   // приоритетная скорость

  // --- Демпинг: снятые метрики (мониторинг) ---
  firstPlacePrice Int?                        // цена 1 места из offer-view
  ourPosition     Int?                        // наше место (1 = первые)
  competitorCount Int?                        // число конкурентов (без ignore и нас)
  lastDumpCheckAt DateTime?                   // когда последний раз проверяли
  lastDumpError   String?                     // текст последней ошибки (или null)

  @@index([autoDownscale])
  @@index([turbo])
}
```

## 5. Движок (расчёт цены)

Файл `lib/kaspi-dumping.ts`.

### 5.1 Снятие конкурентов

```ts
async function fetchOffers(pid: string, cityId: string): Promise<Offer[]>
```
- POST на offer-view (UA + Referer + body как в §1.1).
- Ретрай 429/5xx с экспонентой (переиспользовать паттерн из `kaspi-api.ts`).
- Таймаут ~8с (как в probe).
- Возвращает массив `{price, merchantId, merchantName, rating, kaspiDelivery}`.

### 5.2 Расчёт целевой цены

```ts
function computeTargetPrice(offer: KaspiOffer, offers: Offer[]): {
  target: number | null,         // null = ничего не менять
  position: number,
  firstPlacePrice: number,
  competitorCount: number,
  status: 'winning' | 'floor' | 'no_competitors' | 'matched' | 'error'
}
```

Алгоритм:
1. **Гард floor**: если `autoDownscale && minPriceTenge == null` → НЕ трогать,
   `status='error'`, lastDumpError='нет мин. цены'. (фильтр «Не хватает мин цены»).
2. Исключить из `offers` свой merchantId (`KASPI_OFFERVIEW_MERCHANT_ID`) и всех из
   `ignoreMerchants`. Оставшиеся = конкуренты.
3. `competitorCount = конкуренты.length`. Если 0:
   - если `autoUpscale && maxPriceTenge` → `target = maxPriceTenge` (вернуть маржу),
     `status='no_competitors'`.
   - иначе ничего не менять.
4. Минимальный конкурент `minComp = min(price конкурентов)`.
5. По стратегии:
   - **BECOME_FIRST**: `desired = minComp - dumpingStep`
   - **MATCH_FIRST**: `desired = minComp`
   - **HOLD_SECOND** (позже): `desired = second_lowest + step` и т.п.
6. **Clamp**: `target = clamp(desired, minPriceTenge, maxPriceTenge ?? desired)`.
   - Если `desired < minPriceTenge` → `target = minPriceTenge`, `status='floor'`
     (мы у дна, дальше не падаем — конкурент может обогнать, это норма).
7. **Автоповышение**: если текущая цена < target (мы ниже, чем нужно) — поднимать
   только если `autoUpscale=true`. Снижать (текущая > target) — только если
   `autoDownscale=true`. Иначе соответствующее изменение не применяем.
8. Если `target == currentPrice` → ничего не менять (но метрики обновить).

> ⚠️ **Защита от ценовой войны**: когда оба у floor — НЕ опускаться ниже floor
> (п.6). Дополнительно: если конкурент уже ≤ нашего floor, оставаться на floor
> и не дёргать цену каждый цикл (анти-флаппинг: менять только если разница > 0).

### 5.3 Применение

- Записать `priceTenge = target` в `KaspiOffer`.
- Записать метрики: `firstPlacePrice`, `ourPosition`, `competitorCount`,
  `lastDumpCheckAt = now`, `lastDumpError = null`.
- Залогировать в `productChangeLog` (old→new цена, причина: стратегия + конкурент).
- Фид (`feed.xml`) отдаст новую цену автоматически — отдельный push не нужен.

## 6. Крон (2 скорости)

Роут `api/cron/kaspi-dumping` (по образцу `api/cron/kaspi-sync`), защита `?secret=CRON_SECRET`.

- Параметр `?mode=normal|turbo`:
  - `normal` (раз в 10-15 мин): офферы где `(autoDownscale OR autoUpscale) AND active AND NOT turbo`.
  - `turbo` (раз в 1-2 мин): те же условия + `turbo=true`.
- Внутри прогона:
  - Глобальный гард `KASPI_DUMPING_ENABLED` (если off — выходим).
  - Для каждого оффера: построить PID → fetchOffers → computeTargetPrice → apply.
  - **Троттлинг**: пауза ~500-800мс между offer-view запросами (не долбить Kaspi).
  - Ошибки по одному товару не валят весь прогон (try/catch на item, пишем lastDumpError).
- Cron на EC2 (пользователь `ubuntu`, как Google Indexing): два crontab-строки на
  оба mode. Альтернатива — внешний планировщик, дёргающий роут.

> На 188 активных товарах с паузой 0.6с прогон ≈ 2 мин. На «турбо» держать только
> десятки ключевых SKU, иначе не уложиться в окно + риск лимита Kaspi.

> ⚠️ **РЕАЛЬНОСТЬ СКОРОСТИ (§1.6.2):** наш крон считает новую цену и пишет
> `priceTenge` хоть каждую минуту, НО **Kaspi перечитывает фид раз в ~60 мин**
> (+ ~15 мин применение). Значит фактический такт демпинга через фид ≈ **час**,
> независимо от частоты крона. Частить крон чаще раза в ~15-30 мин бессмысленно
> (фид всё равно перечитается раз в час). «Турбо» в нашей реализации = более
> приоритетный/частый пересчёт *внутри* часового окна, а не суб-минутный демпинг.
> Суб-минутная скорость как у AlgaTop требует кабинетного канала, который мы НЕ
> используем. В UI честно: «цена применится в течение часа-полутора».

## 8. Маржа и комиссия Kaspi

> **РЕШЕНИЕ ВЛАДЕЛЬЦА (2026-05-30):** комиссия Kaspi ≈ **41%**. Расчёт минимальной
> безубыточной цены продажи: **`floorBezubytok = costPrice × 1.41`**. То есть чтобы
> «выйти в ноль» по комиссии, цена продажи должна быть не ниже закуп × 1.41.

- Единая ставка в `AppSetting` `KASPI_COMMISSION_MULT = 1.41` (множитель), чтобы можно
  было править без деплоя. Базовый floor по умолчанию = `costPrice × KASPI_COMMISSION_MULT`.
- **Маржа в UI** (под каждой ценой, как в AlgaTop): `маржа(цена) = цена − costPrice × 1.41`.
  - маржа > 0 (зелёная) — в плюсе; маржа < 0 (красная) — продаём в убыток по комиссии.
  - В процентах: `маржа% = (цена − costPrice × 1.41) / цена × 100`.
- **Авто-floor** (bulk-кнопка): `minPriceTenge = round(costPrice × 1.41)` для выбранных
  товаров с заполненным costPrice. Опционально + желаемая наценка сверху (параметр в
  диалоге): `minPriceTenge = round(costPrice × 1.41 × (1 + наценка%/100))`.
- Если `costPrice` пуст — маржа не показывается («—»), авто-floor для товара не считается.

> Примечание: 41% — упрощённая единая ставка по решению владельца (реальная комиссия
> Kaspi категорийная, но для нашего ассортимента берём 41% как рабочее приближение).
> Множитель ×1.41 правится в `AppSetting` без кода.

## 8.1 Антидемпинг-барьеры Kaspi (CONFIRMED, учесть в логике)

Официальные ограничения Kaspi, которые конфликтуют с агрессивным демпингом
(источник: guide.kaspi.kz/.../goods/discount). Влияют на стратегию авто-повышения:

- **Скидочная плашка ломается частыми колебаниями цены.** Чтобы Kaspi показал
  «скидку», цена за последние **7 дней** должна была меняться **не более чем на 3%**.
  Демпинг-бот, дёргающий цену каждый час, делает плашку «скидка» недоступной.
- **Авто-повышение гасит скидку:** если поднять цену выше «дисконтной базы» (цена до
  скидки) — плашка слетает (действует 30 дней). Логику `autoUpscale` синхронизировать
  с дисконт-механикой, иначе теряется визуальный буст.
- **Размер скидки** Kaspi считает сам (10–80% от цены до скидки) — бот не задаёт %.
- 🔑 **Вывод:** для товаров с активной скидкой — либо НЕ включать частый демпинг
  (держать стабильную цену ради плашки), либо принять, что скидки не будет. Это
  настройка на товар; на старте можно игнорировать (скидок у нас немного), но
  заложить в риски и не удивляться, если плашка «скидка» пропадёт после демпинга.

## 9. Риски и открытые вопросы

1. **Блокировка offer-view с EC2 — ПОДТВЕРДИЛАСЬ (2026-05-30).** С дата-центр IP
   нашего прода (`13.51.198.130`, AWS) Kaspi отдаёт offer-view **405 Not Allowed**,
   а HTML-карточку — **429**. С резидентного IP (локалка) — 200. РЕШЕНИЕ: цены
   снимает **внешний воркер на резидентном IP** (`scripts/kaspi-dumping-worker.mjs`),
   сервер сам в Kaspi НЕ ходит — только считает и применяет (эндпоинты
   `api/admin/kaspi-dumping/tasks` + `ingest`). Полный цикл протестирован
   end-to-end (869→692₸, лог в productChangeLog). См. §6.1.
   **Прод-крон `api/cron/kaspi-dumping` бесполезен (его fetchOffers упрётся в 405) —
   использовать воркер, не крон.**
2. **Скорость через фид = ~60–75 мин** (§1.6.2, CONFIRMED). Суб-минутный демпинг
   недоступен без кабинетного канала. Не обещать в UI секунды.
3. **Ценовая война** с другим репрайсером — защита floor + анти-флаппинг (§5.2).
4. **costPrice пуст** на всех товарах — авто-floor (×1.41) не сработает, пока не
   зальём закуп. На старте floor вручную/Excel.
5. **merchantId рассинхрон** (30383258 в выдаче vs 30233309 в фиде, §1.6.5) —
   зафиксировать `30383258` как «свой» в env, не исключить себя ошибочно.
6. **Антидемпинг-скидки** (§8.1) — частый демпинг ломает плашку «скидка» (3% за 7 дней).
7. **Бонусы конкурентов** (§1.6.3) — чистый ценовой демпинг не гарантирует топ, если у
   конкурента бонусная программа. Детектировать, если видно в выдаче.
8. **Нет лога при смене priceTenge** — текущие PATCH/bulk над `kaspi-offers` НЕ пишут
   `productChangeLog`. Демпинг-изменения цены будут невидимы в истории, пока не добавим
   логирование (field `kaspi_dumping`, guard от NaN). **Добавить в Фазе 1.**
9. **PID не у всех офферов** — `kaspiPidFromSku` работает только для SKU вида `PID_xxx`;
   для «голых» нужен `KaspiCatalogEntry.kaspiProductId`. Проверить покрытие на 188 товарах.
6. **Рейтинг для MATCH_FIRST** — стратегия «равная цена» выигрывает только если наш
   рейтинг ≥ конкурента; offer-view отдаёт `merchantRating`, можно проверять.
7. **Снятые товары (387)** — бот работает только по `active`, снятые не трогаем.
8. **Не повторять схему AlgaTop с логином в кабинет** (§2.0) — нам не нужен вход
   по email/SMS и хранение пароля продавца. Цену меняем своим фидом, товары/закуп
   уже в БД, конкурентов берём из публичного offer-view. Если при реализации
   возникнет соблазн «а давайте логиниться в кабинет ради X» — сначала проверить,
   не решается ли X через Merchant API / фид / нашу БД.
