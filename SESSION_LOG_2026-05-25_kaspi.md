## 2. Структура SKU карточек (ВАЖНО)

В карточках Kaspi (`KaspiCatalogEntry.kaspiSku`) три вида SKU:
1. **Короткий артикул** (`22`, `685`, ≤~2500) = наш `Product.sku`. Привязка к
   товару — авто по артикулу. Но Kaspi-id (PID) из него НЕ достать.
2. **Длинный `цифры_цифры`** (`124086077_810638204`): первая часть = реальный
   **Kaspi product-id**, вторая = id оффера. Из него строится рабочая ссылка.
3. **Большое отдельное число** (`763167519`): мусорный/чужой SKU, НЕ Kaspi-id.

**Прямая ссылка на карточку:** `https://kaspi.kz/shop/p/-<PID>/?c=750000000`
(slug не нужен — Kaspi сам редиректит по PID). Хелпер: `src/lib/kaspi-url.ts`
(`kaspiPidFromSku`, `kaspiUrlFromPid`, `kaspiLinkFromSku`).

**Правило валидности PID для ссылки:**
- SKU с `_` → PID = часть до `_` → ВАЛИДНО.
- PID найден по совпадению названия (pid ≠ sku) → валидно.
- `pid == sku` (короткий артикул/большое число взято как PID) → **МУСОР, → 404**.

## 3. ⚠️ Проверка Kaspi-ссылок — только с локальной машины

Kaspi.kz **блокирует серверные запросы** (с прод-EC2 все ссылки отдают 404 /
капчу). Проверять ссылки `curl`-ом **с локального Mac**, не с сервера. Иначе
ложно посчитаешь все ссылки нерабочими. Надёжный критерий без запросов:
`pid==sku && нет "_"` = мусор.

## 5. Что сделано в КОДЕ (на постоянку, задеплоено)

- **Авто-URL при импорте**: парсер `kaspi-catalog/import` сразу проставляет
  PID + ссылку для длинных SKU (ручные URL не перезаписывает, только пустые).
- **Кнопка «Get URL»** в `/admin/kaspi`: строит ссылку из длинного SKU одним
  кликом (`PATCH kaspi-catalog/[id]` с `{action:'geturl'}`). Для коротких
  артикулов кнопка не показывается.
- **2 аварийных тумблера** (`/admin/kaspi`, key-value `AppSetting`, на лету без
  рестарта): «Товары на Kaspi (фид)» (off → feed.xml пустой) и «Блоки Купить на
  Kaspi на сайте» (off → getKaspiBuyData → null). Не мутируют per-offer данные →
  включение полностью восстанавливает.
- **Пагинация** каталога `/admin/kaspi` (500/стр).
